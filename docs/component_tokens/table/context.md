# Создание компонентных токенов для таблицы: подход по контексту

## Задача

Я дизайнер и у меня задача создать компонент "таблица". Мне нужно определить все токены, которые будут использоваться для стилизации этого компонента, используя контекстный подход.

## Выбор подхода

Я выбираю подход по контексту, потому что мне важно понимать, **где и как** используется компонент. Таблица — это конкретный компонент, и я создаю токены специально для него, ссылаясь напрямую на базовый слой.

## Почему семантический слой не нужен?

При контекстном подходе семантический слой **не нужен**, потому что:

1. **Нет переиспользования**: Токены таблицы используются только в таблице. Использовать `color.table.header` в других компонентах — это плохая практика.

2. **1:1 дублирование**: Если семантический слой специфичен для компонента (например, `semantic_context.color.table.*`), он просто дублирует компонентный слой без добавления абстракции.

3. **Прямая связь с базой**: Компонентные токены могут напрямую ссылаться на базовый слой, что проще и понятнее.

## Процесс создания токенов

### Шаг 1: Анализ структуры таблицы

Сначала я думаю о том, из каких частей состоит таблица:
- Заголовок (header) — показывает названия колонок
- Строки (row) — содержат данные
- Ячейки (cell) — содержат конкретные значения
- Состояния: default, hover, selected

### Шаг 2: Выбор базовых токенов для заголовка

Для заголовка таблицы:
- `base.color.gray.50` — фон заголовка
- `base.color.gray.900` — текст заголовка
- `base.color.blue.500` — для sorted колонки
- `base.color.gray.300` — граница заголовка

### Шаг 3: Выбор токенов для строк

Строки таблицы:
- `base.color.white` — основной фон строки
- `base.color.gray.50` — для hover
- `base.color.blue.500` — для selected строки
- `base.color.blue.600` — для selected строки при hover
- `base.color.gray.900` — текст строки
- `base.color.white` — текст на selected строке
- `base.color.gray.200` — граница строки

### Шаг 4: Выбор токенов для ячеек

Ячейки содержат данные:
- `base.spacing.md` для горизонтального padding
- `base.spacing.sm` для вертикального padding
- `base.color.gray.900` — основной текст ячейки
- `base.color.gray.600` — вторичный текст

### Шаг 5: Typography для таблицы

Для типографики:
- Заголовок: `base.typography.font_size.base`, `base.typography.font_weight.medium`
- Тело: `base.typography.font_size.base`, `base.typography.font_weight.regular`

### Шаг 6: Spacing

- `base.spacing.xs` для row_gap — строки должны быть плотными
- `base.spacing.sm` для cell_gap

## Примеры использования

```json
{
  "table.header.background.default": "{base.color.gray.50}",
  "table.row.background.hover": "{base.color.gray.50}",
  "table.row.background.selected": "{base.color.blue.500}",
  "table.cell.padding.x": "{base.spacing.md}"
}
```

## Преимущества подхода

1. **Простота**: Прямая связь с базовым слоем
2. **Понятность**: Ясно видно, какие базовые значения используются
3. **Быстрота**: Не нужно создавать семантический слой
4. **Специфичность**: Токены максимально специфичны для компонента

## Недостатки подхода

1. **Нет абстракции**: Если нужно изменить цвет всех таблиц, нужно менять в компонентных токенах
2. **Сложнее White Label**: Для разных брендов нужно менять компонентные токены

## Размышления

Когда я создаю компонентные токены для таблицы при контекстном подходе, я мыслю так: "Это таблица, значит мне нужны токены специально для таблицы." Я выбираю базовые токены напрямую, без промежуточного семантического слоя.

Особенно важно для таблицы правильно обработать состояния hover и selected — это ключевые функциональные состояния, которые помогают пользователю ориентироваться в данных.
