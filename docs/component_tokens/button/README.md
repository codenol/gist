# Создание компонентных токенов для кнопки

## Задача

Я дизайнер и у меня задача создать компонент "кнопка". Мне нужно определить все токены, которые будут использоваться для стилизации этого компонента. В этой документации я опишу процесс создания токенов для всех четырех подходов к организации семантических токенов.

## Общие части для всех подходов

### Структура компонента

Все подходы используют одинаковую структуру кнопки:
- **Варианты**: primary, secondary
- **Состояния**: default, hover, active, disabled
- **Размеры**: small, medium, large
- **Элементы**: background, text, border, padding, typography, border_radius, shadow

### Основные токены

Независимо от подхода, кнопка требует следующие токены:
- `button.primary.background.*` — фон primary кнопки для всех состояний
- `button.primary.text.*` — текст primary кнопки
- `button.primary.border.*` — граница primary кнопки
- `button.secondary.*` — аналогично для secondary кнопки
- `button.padding.*` — отступы
- `button.typography.*` — типографика
- `button.border_radius` — скругление
- `button.size.*` — размеры

## Различия между подходами

### 1. Подход по функциональности

**Философия**: Думаю о том, **что делает** каждый элемент.

**Ключевые семантические токены**:
- `semantic_functional.color.action.primary` — для primary кнопки (это действие)
- `semantic_functional.color.background.secondary` — для secondary кнопки (это фон)
- `semantic_functional.color.text.inverse` — для текста на цветном фоне
- `semantic_functional.spacing.padding.medium` — стандартный отступ для интерактивных элементов

**Особенности мышления**:
- Primary кнопка = действие → `color.action.*`
- Secondary кнопка = фон → `color.background.*`
- Текст = функциональная роль → `color.text.*`

**Преимущества**: Интуитивность, легко понять назначение каждого токена.

**Недостатки**: Может быть сложно для White Label, так как функциональность привязана к конкретным значениям.

### 2. Подход по ролям

**Философия**: Думаю о **роли** каждого элемента в системе дизайна.

**Ключевые семантические токены**:
- `semantic_roles.color.brand.primary` — для primary кнопки (брендовый элемент)
- `semantic_roles.color.surface.secondary` — для secondary кнопки (нейтральная поверхность)
- `semantic_roles.color.neutral.text_primary` — нейтральный текст
- `semantic_roles.spacing.role.regular` — роль размера (не просто значение)

**Особенности мышления**:
- Primary кнопка = бренд → `color.brand.*`
- Secondary кнопка = нейтральная поверхность → `color.surface.*` + `color.neutral.*`
- Spacing = роли (compact, regular, spacious), а не конкретные значения

**Преимущества**: Идеально для White Label — роли остаются, значения меняются. Системность мышления.

**Недостатки**: Требует понимания системы ролей, может быть менее интуитивным для новичков.

### 3. Подход по иерархии

**Философия**: Думаю о **уровне важности** каждого элемента.

**Ключевые семантические токены**:
- `semantic_hierarchy.color.level_1.action` — для primary кнопки (самый важный уровень)
- `semantic_hierarchy.color.level_2.background` — для secondary кнопки (менее важный)
- `semantic_hierarchy.color.level_3.*` — для disabled состояний (наименее важный)
- `semantic_hierarchy.spacing.level_2.padding` — средний уровень отступов

**Особенности мышления**:
- Primary кнопка = level_1 → `color.level_1.*`
- Secondary кнопка = level_2 → `color.level_2.*`
- Disabled = level_3 → `color.level_3.*`
- Typography level_3 для основного текста (не заголовок)

**Преимущества**: Четкая визуальная иерархия, легко масштабировать (добавить level_4, level_5).

**Недостатки**: Может быть сложно определить, какой уровень использовать для новых компонентов.

### 4. Подход по контексту

**Философия**: Использую готовые токены из контекста компонента.

**Ключевые семантические токены**:
- `semantic_context.color.button.primary.*` — все состояния уже определены!
- `semantic_context.spacing.button.*` — все spacing уже определены!
- `semantic_context.typography.button.*` — вся типографика уже определена!

**Особенности мышления**:
- Просто мапплю компонентные токены на контекстные
- Не нужно думать о функциональности, ролях или иерархии
- Все уже готово в контексте кнопки

**Преимущества**: Максимальная простота и скорость. Специфичность для компонента.

**Недостатки**: Требует, чтобы семантический слой уже содержал контекстные токены. Менее гибкий для новых компонентов.

## Сравнительная таблица

| Аспект | Функциональность | Роли | Иерархия | Контекст |
|--------|------------------|------|----------|----------|
| **Primary фон** | `action.primary` | `brand.primary` | `level_1.action` | `button.primary.background` |
| **Secondary фон** | `background.secondary` | `surface.secondary` | `level_2.background` | `button.secondary.background` |
| **Текст primary** | `text.inverse` | `surface.primary` | `level_1.text` | `button.primary.text` |
| **Spacing** | `padding.medium` | `role.regular` | `level_2.padding` | `button.padding_x` |
| **Сложность** | Низкая | Средняя | Средняя | Очень низкая |
| **White Label** | Сложно | Идеально | Средне | Сложно |
| **Масштабируемость** | Средняя | Высокая | Высокая | Низкая |

## Рекомендации по выбору подхода

- **Для быстрого прототипирования**: Контекст
- **Для White Label решений**: Роли
- **Для сложных интерфейсов с иерархией**: Иерархия
- **Для интуитивной работы**: Функциональность
